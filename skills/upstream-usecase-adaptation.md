# 上游用例适配流程

本文档记录从英文上游仓库 [awesome-openclaw-usecases](https://github.com/hesamsheikh/awesome-openclaw-usecases) 同步、研究和适配用例到本中文仓库的完整流程。面向项目维护者和 Claude Code 使用。

---

## 总览

整个流程分三步：

1. **对比上下游差异** — 找出英文有但中文没有的用例
2. **逐个用例深度研究** — 获取原版、评估适配性、撰写中文版本
3. **提交与 Review** — 独立分支、独立 PR、更新 README

---

## 第一步：对比上下游差异

### 操作方法

1. 抓取英文上游仓库的 README，提取用例列表
2. 抓取英文上游仓库 `usecases/` 目录，列出所有 `.md` 文件
3. 列出本地中文仓库 `usecases/` 下的所有 `.md` 文件
4. 逐一比对，找出差异：
   - 英文有、中文没有的用例（待新增）
   - 英文已更新、中文版本落后的用例（待同步）
   - 中文独创用例（`cn-` 前缀，无需比对）

### 输出格式

建议以表格形式记录差异：

| 英文文件名 | 中文对应 | 状态 |
|-----------|---------|------|
| `meeting-notes-action-items.md` | `meeting-notes-action-items.md` | 已同步 |
| `podcast-production-pipeline.md` | 无 | 待新增 |
| `pre-build-idea-validator.md` | 无 | 待新增 |

---

## 第二步：逐个用例深度研究

每个用例独立完成以下四个环节。不要批量翻译，逐个理解后再动手。

### 2.1 获取英文原版

从上游仓库抓取原始 `.md` 文件，完整理解以下内容：

- **痛点和使用场景** — 这个用例解决什么现实问题
- **核心功能** — 具体能做什么
- **所需 skills** — MCP servers、社区 skills、第三方集成
- **所需额外工具** — API、平台账号、硬件等
- **配置步骤和提示词** — 端到端的操作流程

### 2.2 国内市场适配性分析

这是最关键的一步，决定用例是否收录以及如何适配。

#### 评估清单

| 评估维度 | 要回答的问题 |
|---------|------------|
| 工具可用性 | 原用例依赖的海外工具/平台在国内是否可用？ |
| 替代方案 | 国内对应替代方案是什么？（如 Google Meet -> 飞书妙记 / 腾讯会议 / 钉钉） |
| 使用习惯 | 国内用户的使用习惯有何差异？ |
| 合规性 | 是否涉及法律合规风险？（如电话外呼需许可证、社交平台封号风险） |
| skill 生态 | 替代方案有没有现成的 OpenClaw skill 可用？ |

#### 适配性判断结论

根据评估结果，给出以下三种判断之一：

- **直接收录** — 原版内容在国内完全可用，翻译即可（如 Todoist、GitHub 等国际通用工具）
- **需要适配** — 核心功能可用但需替换部分工具或增加国内方案（大部分用例属于这类）
- **不建议收录** — 核心服务在国内不可行且无合理替代（记录原因，供未来重新评估参考）

### 2.3 中文版本撰写

#### 文件命名

- 翻译自上游的用例：保持英文原文件名不变（如 `meeting-notes-action-items.md`）
- 中国独创用例：使用 `cn-` 前缀（如 `cn-feishu-ai-assistant.md`）

#### 内容规范

1. **标题**：使用中文标题
2. **适配提示**：如有国内适配内容，在标题下方加一行提示：
   ```
   > 含国内适配：飞书妙记 / 腾讯会议 / 钉钉
   ```
3. **正文**：描述、痛点、功能说明等使用中文
4. **提示词**：保留英文原文（OpenClaw 使用英文效果最佳），上方用中文说明其作用
5. **代码块**：代码和命令保留英文，注释翻译为中文
6. **技术术语**：首次出现时添加括号注释（如 webhook（网络钩子））
7. **格式**：与仓库现有文件保持一致（参考 `usecases/` 下的已有文件）

#### 章节结构参考

```
# 中文标题

> 含国内适配：XX / YY / ZZ（如有）

[痛点描述段落]

## 痛点（可选，部分用例将痛点融入开头段落）

## 它能做什么

## 所需技能

## 如何设置 / 设置方法

## 关键洞察 / 实用建议

## 中国用户适配（如需要）

## 相关链接
```

### 2.4 中国用户适配章节

如果用例需要国内适配，添加 `## 中国用户适配` 章节，包含以下内容：

#### 必须包含

- **工具替代映射表**：

  | 原版工具 | 国内替代 | OpenClaw 技能 |
  |---------|---------|---------------|
  | Otter.ai | 飞书妙记 | feishu-doc |
  | Slack | 飞书群聊 | 飞书渠道插件 |

- **分平台方案**：如涉及多个国内平台，按平台分节说明（飞书用户方案、钉钉用户方案、腾讯会议用户方案等）

#### 建议包含

- **安全提醒**：凭证环境变量、敏感信息处理、平台风控提醒
- **前置条件**：引用已有的 IM 接入用例（如"已完成飞书 AI 助手的基础接入"）
- **实用建议**：先用最简单的方式验证，确认效果后再自动化

#### 特殊情况

如果原方案在国内完全不可行：
- 如实说明原因
- 推荐替代思路（如果有的话）
- 不强行适配，避免误导用户

## 第三步：提交与 Review

### 分支与 PR 规范

- 每个用例（或一批紧密相关的用例）使用独立分支
- 分支命名：`feat/xxx`、`fix/xxx`、`improve/xxx`
- PR 标题使用中性描述（如"新增会议纪要用例"而非"翻译 meeting notes 用例"）
- 不在 PR 中暴露项目运营策略或内部讨论

### README 更新

- 将新用例添加到 README 的对应分类表格中
- 如有国内适配内容，同时在"中国特色用例"表格中列出
- 更新顶部的用例计数徽章
- README 更新可以和用例放在同一个 PR 中

### 提交检查清单

- [ ] 用例文件放在 `usecases/` 目录下
- [ ] 文件名符合命名规范
- [ ] 提示词保留英文原文
- [ ] 描述性内容为中文
- [ ] 格式与现有文件一致
- [ ] 适配章节包含工具映射表（如需要）
- [ ] 安全提醒已添加（如涉及凭证或敏感操作）
- [ ] README 已更新（目录条目 + 计数徽章）
- [ ] 没有硬编码 API 密钥

## 淘汰标准参考

以下情况不建议收录，记录原因供未来重新评估：

| 淘汰原因 | 示例 |
|---------|------|
| 核心服务在国内完全不可用且无合理替代 | clawr.ing 电话通知（国内无同等服务） |
| 涉及封号风险的灰色操作 | 个人微信协议方案 |
| 合规风险过高 | 个人用户自动外呼需呼叫中心许可证 |
| 中国用户使用场景不匹配 | 食谱转购物清单（菜市场/社区团购 vs 超市模式） |

注意：淘汰判断不是永久的。随着国内工具生态和 OpenClaw skill 生态的发展，之前不可行的用例可能变得可行，建议定期重新评估。

## Claude Code 使用说明

当使用 Claude Code 执行此流程时，以下是推荐的工具使用方式：

### 第一步：对比差异

```
# 抓取上游 README
WebFetch: https://github.com/hesamsheikh/awesome-openclaw-usecases

# 抓取上游 usecases 目录
WebFetch: https://github.com/hesamsheikh/awesome-openclaw-usecases/tree/main/usecases

# 列出本地用例
Glob: usecases/*.md
```

### 第二步：研究用例

```
# 抓取英文原版
WebFetch: https://raw.githubusercontent.com/hesamsheikh/awesome-openclaw-usecases/main/usecases/xxx.md

# 搜索国内替代工具和 skill
WebSearch: "OpenClaw [工具名] skill"
WebSearch: "[国内工具] API 接入"

# 查看已有适配用例作为参考
Read: usecases/meeting-notes-action-items.md
Read: usecases/cn-feishu-ai-assistant.md
```

### 第三步：提交

```bash
# 创建分支
git checkout -b feat/xxx

# 提交用例
git add usecases/xxx.md
git commit -m "新增 xxx 用例"

# 更新 README 后提交
git add README.md
git commit -m "更新 README：新增 xxx 用例"

# 推送并创建 PR
git push -u origin feat/xxx
gh pr create --title "新增 xxx 用例" --base feat/cn-finance-usecases
```
